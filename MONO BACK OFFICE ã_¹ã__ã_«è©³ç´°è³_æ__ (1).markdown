## 1. はじめに

本資料は、株式会社MONO-MARTが運用するZOZO分析Webアプリケーション「MONO BACK OFFICE」およびその関連業務で使用されている全12種類のAIエージェントスキルの詳細を包括的にまとめたものである。本資料の目的は、他のAIエージェント（Claude、ChatGPT等）がシステム開発や業務自動化を行う際に、既存のスキル体系を正確に理解し、同等の機能を再現・拡張できるようにすることにある。

各スキルは、ZOZOTOWNにおけるEC運営業務（在庫管理、発注、広告、セール、MD計画など）を自動化・効率化するために設計されており、データの取得元、処理ロジック、出力形式、技術的な制約事項に至るまで、実装に必要な情報を網羅的に記載している。

## 2. 発注管理スキル

### 2.1 zozo-repeat-order-excel — リピート発注管理表 Google Spreadsheet 自動生成

#### 2.1.1 目的・用途

ZOZOで販売中の特定品番について、リピート発注（追加発注）の意思決定を支援するための管理表をGoogle Spreadsheetとして自動生成するスキルである。品番、発注総数、要望を入力として受け取り、展開SKU、在庫、注文、入荷予定、予約、カラーマスタなど複数のデータソースから情報を収集・統合する。最終的に、10日刻みの売上実績、SKUごとの最適な発注配分、商品画像を自動挿入した包括的なGoogle Spreadsheetを出力する。

**トリガー例**: 「リピート発注表を作って」「品番sc841のリピート発注」「発注管理表の作成」

#### 2.1.2 入力パラメータ

| パラメータ | 必須 | 説明 | 例 |
|---|---|---|---|
| 品番 | 必須 | 対象のブランド品番 | sc841 |
| リピート発注総数 | 必須 | 追加発注する総数量 | 6000 |
| 対象カラー | 任意 | 「既存色のみ」「全色」「指定色のみ」から選択（デフォルト: 既存色のみ） | 既存色のみ |
| 要望・メモ | 任意 | 自由記述の補足情報 | Lサイズ多めに |

#### 2.1.3 出力

Google Spreadsheet形式のリピート発注管理表が生成される。以下の5シートで構成される。

| シート名 | 内容 |
|---|---|
| 発注表 | SKUごとの基本情報、在庫状況、SKUバランス配分に基づく発注数、10日刻み売上実績、IMAGE関数による商品画像 |
| 売上実績 | 注文生データから集計した10日刻みの売上実績（T列以降にヘッダー、A-S列はヘッダーなし） |
| 在庫分析貼り付け | 在庫分析CSVの生データ（ヘッダー行なし） |
| 予約管理表 | 予約一覧CSVから集計した予約未処理数 |
| 予約一覧 | 予約一覧CSVの生データ |

#### 2.1.4 データソース

本スキルは以下の6つのデータソースとテンプレートファイルを使用する。

| データソース | ファイル名/形式 | エンコーディング | 用途 |
|---|---|---|---|
| 展開SKU CSV | goods_cs.csv（約170MB） | cp932 | 品番からショップ名・商品コードを特定 |
| 在庫分析CSV | 在庫状況({ショップ名}).csv | cp932 | 在庫数、販売数、お気に入り登録数 |
| 注文生データ | order_YYYYMM.csv | UTF-8 | 10日刻み売上実績の算出 |
| 入荷予定 | Googleスプレッドシート | - | ブランド別入荷予定データ |
| 予約管理一覧CSV | reserve_list_{ショップ名}.csv | cp932 | 予約未処理数の取得 |
| ZOZOカラーマスタ | color_master.csv | - | カラー名とカラーコードの対応表 |
| テンプレート | 10日刻み用.xlsm | - | Spreadsheetの元フォーマット（Google Drive ID: 1SMO_tV2Cs1kRqr8s9EBCftSoGV4PPiyS） |

**データ処理上の注意点**: 展開SKU CSVは約170MBと巨大であるため、`grep`で事前に対象品番を絞り込んでから読み込む。CSVの列名に含まれる先頭スペースを`.strip()`で除去する必要がある。データ行の末尾に余分なカンマが含まれるケースにも対応が必要である。デプロイ環境では、巨大CSVは事前にJSON形式に変換してS3 CDNに配置し、オンデマンドで読み込む方式を採用している。

#### 2.1.5 SKUバランス配分ロジック

発注総数を各SKUに最適配分するアルゴリズムは以下の通りである。

1. **基本構成比の算出**: 直近30日間の販売実績から、各SKU（カラー×サイズ）の販売構成比を算出する
2. **在庫日数による調整**: SKUごとの在庫日数に応じて調整係数を乗算する
   - 在庫完売（在庫0）: 1.5倍（需要が高いと判断）
   - 在庫日数60日超: 0.7倍（過剰在庫と判断）
3. **端数調整**: 調整後の配分値を丸め、合計が発注総数と一致するように端数を調整する

#### 2.1.6 商品画像の自動挿入

ZOZOカラーマスタを基に、以下のルールで商品画像URLを生成し、Google SheetsのIMAGE関数で表示する。

**画像URL生成ルール**:
```
https://o.imgz.jp/{商品コード末尾3桁}/{商品コード}/{商品コード}b_{カラーコード}_d.jpg
```

**IMAGE関数の設定**:
```
=IMAGE("url", 4, 150, 120)
```
モード4（カスタムサイズ）、高さ150px、幅120pxで表示する。カラー別に2列間隔（B, D, F列...）で配置し、カラー名行とその下の画像行の2行構成とする。画像行の高さは160pxに設定する。

**SKUの並び順**: カラー名昇順でグループ化し、各カラー内ではサイズ昇順（FREE → S → M → L → XL → XXL → 3XL → 4XL）で表記する。

#### 2.1.7 技術的な実装（MONO BACK OFFICE組み込み）

MONO BACK OFFICEへの組み込みでは、以下のモジュール構成でTypeScriptによるバックエンドAPIとして実装されている。

| ファイル | 役割 |
|---|---|
| server/repeat-order/api.ts | APIエンドポイント（POST /api/repeat-order） |
| server/repeat-order/data-loader.ts | S3 CDN/Google APIからのデータ取得 |
| server/repeat-order/sku-balance.ts | SKUバランス配分ロジック |
| server/repeat-order/sheets-writer.ts | Google Sheets API書き込み・画像挿入 |
| server/repeat-order/config.ts | データソースURL定数（DATA_URLS） |

デプロイ環境ではGoogle Sheets APIをサービスアカウント方式で呼び出す。環境変数`GOOGLE_SERVICE_ACCOUNT_KEY`にJSONキーを設定する。

#### 2.1.8 注意事項

- IMAGE関数で表示される商品画像は、初回表示時にGoogle Sheets上で「Allow access」ボタンの手動クリックが必要である
- デプロイ環境とサンドボックス環境ではデータアクセス方式が異なる（デプロイ: S3 CDN + Google API、サンドボックス: ローカルファイル + gws CLI）
- テンプレートExcelファイル(.xlsm)にはマクロが含まれるが、自動生成されるファイルはマクロなしの形式となる

### 2.2 zozo-repeat-simulation — リピート推奨数量シミュレーション

#### 2.2.1 目的・用途

ZOZOの月別実績データとカテゴリ別季節係数を用いて、品番ごとのリピート推奨数量（追加発注数）を算出するスキルである。実績期間に応じた予測方式の切り替え、直近粗利率低下ペナルティ、ピーク月選択による需要算出ロジックを含んでおり、2つのパターン（ペナルティなし/強化版）で推奨数量を比較し、発注判断を支援する。

**トリガー例**: 「リピート発注のシミュレーションをお願いします」

#### 2.2.2 入力パラメータ

| パラメータ | 必須 | 説明 |
|---|---|---|
| item_id | 必須 | 品番情報 |
| category | 必須 | ZOZO子カテゴリ名 |
| current_inv | 必須 | 現在庫数 |
| monthly_sales | 必須 | 月別販売実績（4月始まりの12要素配列） |
| gm_score | 必須 | 粗利率スコア |
| sales_score | 必須 | 販売数スコア |
| season_score | 必須 | 季節安定性スコア |
| cross_score | 必須 | 併売率スコア |
| gm_rate | 必須 | 粗利率 |
| gm_change | 必須 | 直近2ヶ月のGM変化率 |

#### 2.2.3 予測アルゴリズム

販売実績の期間と規模に応じて、以下の4つの予測方式を動的に切り替える。

| 条件 | 予測方式 | 説明 |
|---|---|---|
| 実績3ヶ月未満 | フラット | 直近月の販売数を一律適用 |
| 実績3-4ヶ月 & 月販1,000着未満 | 季節係数単純平均 | カテゴリ季節係数で補正した平均 |
| 実績4-6ヶ月 | 中央値 | 月別販売数の中央値を使用 |
| 実績6ヶ月以上 & 月販1,000着以上 | 直近重み付き平均 | 直近月に高い重みを付与 |

#### 2.2.4 総合スコアと在庫持ち月数

各種スコアを以下の重みで加算して総合スコアを算出し、それに基づいて在庫持ち月数を決定する。

| スコア項目 | 重み |
|---|---|
| 粗利率スコア (gm_score) | 40% |
| 販売数スコア (sales_score) | 25% |
| 季節安定性スコア (season_score) | 20% |
| 併売率スコア (cross_score) | 15% |

総合スコアに応じた在庫持ち月数は5段階（2.0ヶ月〜5.0ヶ月）で設定される。

#### 2.2.5 ペナルティ機構

- **GM低下ペナルティ**: 直近2ヶ月のGMが-2pp以上低下した場合に発動
- **粗利率絶対値ペナルティ**: 粗利率が30%未満の場合に発動

最終的なリピート推奨数量は以下の式で計算される。

```
推奨数量 = max(0, ピーク月需要合計 - 7月末予測在庫)
```

「ピーク月選択方式」により、季節係数が高い月から優先的に需要を合計する。

#### 2.2.6 カテゴリ別季節係数

`references/monthly_coefficients.json`に以下の12カテゴリの月別販売季節係数が定義されている。4月から翌年3月までの12要素の浮動小数点数配列で、52週売上データから算出されたものである。

対応カテゴリ: Tシャツ/カットソー、シャツ/ブラウス、セットアップ、スラックス、デニムパンツ、テーラードジャケット、スウェット、パーカー、ブルゾン、デニムジャケット、モッズコート、トレンチコート

#### 2.2.7 注意事項

- 月別販売実績は4月始まりの12要素配列である必要がある
- 季節係数ファイルで定義されていないカテゴリは計算できない。新カテゴリ対応にはJSONファイルの追記が必要
- 予測ロジックは品番の販売実績期間と月間販売数によって動的に切り替わる

## 3. 併売分析スキル

### 3.1 zozo-same-item-cross-buy — 同一品番併売分析

#### 3.1.1 目的・用途

ZOZOの併売データCSVを用いて、特定ブランド品番の「同一品番併売」（色違い・サイズ違い）を分析するスキルである。同一品番併売率、色の組み合わせ傾向、カテゴリ平均との比較、リピート発注の優先度を評価する「リピートスコア」の算出を行い、カラー展開の拡張提案を行う。分析結果はPDFレポート、Webアプリ用JSON、またはExcel形式で出力可能である。

**トリガー例**: 「同一品番併売分析をお願いします」

#### 3.1.2 入力・出力

**入力**: 併売データCSVファイルのパス（必須）、分析対象のブランド品番（必須）、比較対象のブランド品番（任意）、ブランドキー（任意）

**出力**: PDFレポート（1枚）、Webアプリ用JSONデータ、発注推奨データを含むExcelファイル(.xlsx)、分析結果サマリCSV

#### 3.1.3 データソース

| データソース | 形式 | 用途 |
|---|---|---|
| ZOZO併売データCSV | Shift_JIS、28カラム | 同一品番の併売パターン分析 |
| 注文データCSV | UTF-8 | 月別販売実績の集計（Google Drive、2025年5月〜2026年3月） |
| 原価データ | Googleスプレッドシート | 粗利率の算出 |
| 現在庫データ | Googleスプレッドシート | 在庫状況の把握 |
| 品番別スコアJSON | CDN (item_scores.json) | 事前集計済みのリピートスコア |
| ダッシュボード商品別実績 | Excel (Google Drive) | UU/PV/CVR等のKPI指標 |

#### 3.1.4 リピートスコア算出ロジック

リピートスコアは以下の4つの指標を重み付け加算して算出される。

| 指標 | 重み | 説明 |
|---|---|---|
| 粗利率 | 40% | 品番の粗利率に基づくスコア |
| 販売数 | 25% | 月間販売数に基づくスコア |
| 季節安定性 | 20% | 月別販売の安定度 |
| 併売率 | 15% | 同一品番の併売発生率 |

スコアに応じて在庫持ち月数が2.0〜5.0ヶ月の範囲で設定される。粗利率低下や絶対値に基づくペナルティ（最大30pt）も適用される。

#### 3.1.5 カラー分類ロジック

カラーは「メイン」「シーズン」「アクセント」の3カテゴリに分類される。分類の優先順位は以下の通りである。

1. **ブランド別パレット優先**: `brand_color_palettes.json`にブランドキー（テイスト別）ごとのカラー分類が定義されている
2. **キーワードフォールバック**: パレットに存在しない場合、MAIN_KW（ブラック、ホワイト等）やSEASON_KW（カーキ、ベージュ等）のキーワードで判定

結論提案ロジックとして、メインカラーの追加推奨またはシーズンカラーの追加推奨が自動生成される。

#### 3.1.6 注意事項

- 併売データCSVはShift_JISエンコーディングの28カラム形式が必須
- ブランドキーが未指定の場合、ショップ名から自動判定される
- 品番別スコアJSON（item_scores.json）が最新でないと推奨値が不正確になる可能性がある

## 4. KPI分析スキル

### 4.1 zozo-product-dashboard — 品番ダッシュボード JSON 生成

#### 4.1.1 目的・用途

ZOZOの商品別実績Excelから品番ダッシュボード用のJSONデータを生成し、MONO BACK OFFICE上で品番別の日次KPI、カテゴリ内順位、トレンド分析などを表示するためのスキルである。

**トリガー例**: 「ダッシュボードデータを更新して」「品番パフォーマンス分析」「カテゴリベンチマーク比較」

#### 4.1.2 入力・出力

**入力**: 商品別実績Excelファイル（`/home/ubuntu/dashboard_product_data.xlsx`、シート名: `商品別実績 (2)`）

**出力**: 品番ダッシュボード用JSONファイル（`/home/ubuntu/dashboard_data.json`）

#### 4.1.3 処理ロジック

PythonスクリプトがExcelを読み込み、品番ごとの日次データとカテゴリごとの統計データを集計して一つのJSONファイルにまとめる。カテゴリ統計を算出する際、UUが100以上のアイテムのみを対象に平均値を計算し、ノイズデータを除去する。該当アイテムがない場合は全アイテムで計算する。お気に入り率とCVR_UUでそれぞれ上位10アイテムをリストアップし、全品番のカテゴリ内順位も算出する。

#### 4.1.4 データ更新フロー

1. ZOZO管理画面から商品別実績Excelをエクスポート
2. `scripts/convert_dashboard_to_json.py`を実行してJSONに変換
3. `manus-upload-file`でCDNにアップロード
4. フロントエンドの`useProductDashboard.ts`内の`DATA_URL`を新しいCDN URLに更新

## 5. 在庫管理スキル

### 5.1 zozo-inventory-forecast — 在庫消化予測・不動在庫分析

#### 5.1.1 目的・用途

ZOZOの52週カテゴリ別売上係数と品番別在庫・日販データを用いて、指定された期日までの在庫消化を予測するスキルである。品番ごとに「在庫が余るか」「消化率は何%か」を算出する。

**トリガー例**: 「5月末までに在庫が余るか」「この品番の消化率は何%か」「在庫消化予測をお願いします」

#### 5.1.2 入力・出力

**入力**:

| パラメータ | 必須 | 説明 |
|---|---|---|
| 52週カテゴリ推移Excel | 必須 | ZOZO全体のカテゴリ(子)別・週別販売実績 |
| 品番別在庫Excel | 必須 | MONO-MART形式の品番別在庫・日別販売データ |
| 予測期限 | 必須 | YYYY-MM-DD形式 |
| 対象シート名 | 任意 | 特定シートのみ処理 |
| 対象品番 | 任意 | 特定品番のみ処理 |
| 日販計算の直近日数 | 任意 | デフォルト14日 |

**出力**: 在庫消化予測結果CSV（ブランド、品番、商品名、カテゴリ(子)、シーズン、現在庫、直近日販、予測販売数、残在庫、消化率(%)、判定）

#### 5.1.3 予測アルゴリズム

1. 品番ごとに直近N日間の平均日販を算出する。日別販売データの末尾にある連続した0を除外して実データ期間を自動検出し、直近N日の平均を使用する。直近の販売実績が0の場合は全期間平均にフォールバックする
2. カテゴリの52週係数を使い、以下の式で未来の日販を補正する

```
予測日販 = 平均日販 × (対象週係数 / 現在週係数)
```

3. 予測期限まで日々の予測販売数を積算し、累計予測販売数を算出する
4. 消化率を以下の式で計算する

```
消化率 = 予測販売数 / 現在庫 × 100
```

**週番号の定義**: `week = min((day_of_year - 1) // 7 + 1, 52)`（1月1日〜1月7日が第1週）

#### 5.1.4 ファイル構成

| ファイル | 役割 |
|---|---|
| scripts/build_coefficients.py | 52週係数JSONの生成 |
| scripts/forecast_inventory.py | 在庫消化予測の実行 |
| scripts/show_coefficients.py | 係数確認ユーティリティ |
| references/data-format.md | データフォーマット定義 |

#### 5.1.5 注意事項

- 52週Excelは35万行を超えるため、読み込みに数分かかる場合がある
- 係数JSONファイルは年度ごとに一度生成すれば再利用可能
- 夏物カテゴリのように3月時点で日販が低くても、係数によって5月以降の需要上昇が予測に反映される

## 6. 顧客分析スキル

### 6.1 zozo-affinity-analysis — 買い回りデータ相性ショップ分析

#### 6.1.1 目的・用途

ZOZO内の買い回りデータ（ブランド別Excelシート）とショップ指標データ（月次Excel）を組み合わせて、各ブランドと相性の良い他社ショップを母数補正済みの相性スコアで特定・ランキングする分析スキルである。単純な買い回り件数だけでは分からない、真に相性の良いショップを明らかにする。

**トリガー例**: 「MONO-MARTの買い回り分析をお願いします」「相性の良いショップを教えて」

#### 6.1.2 入力・出力

**入力**: 買い回りデータExcel（ブランド別シート）、ショップ指標データExcel（月次）、分析対象ブランド（任意、デフォルトは全ブランド）

**出力**: 相性分析レポート.xlsx（以下のシートを含む）
- 複数ブランド相性良TOP30
- 全ブランド横断TOP50
- 各ブランド別TOP20シート

#### 6.1.3 相性スコア算出式

```
相性スコア(%) = (買い回り件数 / ショップ購入者数) × 100
```

- **買い回り件数**: 買い回りデータのショップ名ごとのレコード数
- **ショップ購入者数**: ショップ指標データの「購入者数」カラムの値

#### 6.1.4 フィルタ条件

| パラメータ | デフォルト値 | 説明 |
|---|---|---|
| min_buyback | 30件 | 最低買い回り件数 |
| min_buyers | 100人 | ショップ最低購入者数 |
| OWN_BRANDS | リスト | 自社ブランド除外リスト |

#### 6.1.5 データ読み込みの注意点

ショップ指標データExcelはヘッダーが2行目にあるため、pandasで読み込む際に`header=None`を指定し、後からカラム名を設定する必要がある。データ結合は「ショップ名」の文字列完全一致で行われるため、名称の揺れがあるとデータが欠落する可能性がある。

## 7. MD計画スキル

### 7.1 zozo-md-plan — ブランド別×カテゴリ別 MD計画自動生成

#### 7.1.1 目的・用途

ZOZO内の買い回りデータ、ショップ指標データ、注文データ、原価データの4種類を統合し、ブランド別×カテゴリ別のMD計画（強化推奨、新規参入、見直しなど）を自動生成するスキルである。外部環境分析、内部環境分析、クロス分析の3ステップを経て、最終的な提案をJSON形式で出力する。

**トリガー例**: 「MD計画分析をお願いします」「カテゴリ戦略の提案」「品揃え最適化」

#### 7.1.2 3ステップの分析プロセス

| ステップ | スクリプト | 内容 |
|---|---|---|
| Step 1: 外部環境分析 | md_plan_step1_external.py | カテゴリ需要、相性ショップランキング |
| Step 2: 内部環境分析 | md_plan_step2_internal.py | 自社カテゴリ別パフォーマンス |
| Step 3: クロス分析 | md_plan_step3_cross.py | MDスコア算出、アクション提案 |

#### 7.1.3 入力データ

| データ | 形式 | 用途 |
|---|---|---|
| 買い回りデータ | Excel (.xlsx) | ブランドごとにシート分割された月次データ |
| ショップ指標データ | Excel (.xls) | 月次のショップ別集計データ |
| 注文データ | CSV | zozo-order-dataスキルで定義される注文生データ |
| 原価データ | CSV | 品番別の原価情報 |

#### 7.1.4 MDスコア算出ロジック

MDスコアは以下の4つの指標を重み付け加算して算出される。

| 指標 | 重み | 基データ |
|---|---|---|
| 需要スコア | 40% | 買い回り件数 |
| 粗利スコア | 25% | 粗利率 |
| 効率スコア | 20% | 型あたり売上 |
| 成長スコア | 15% | プロパー比率 |

#### 7.1.5 アクション判定テーブル

MDスコアと自社でのカテゴリ展開状況を組み合わせて、以下のアクションを判定する。

| MDスコア | 自社展開あり | 自社展開なし |
|---|---|---|
| 高 | 強化推奨 | 参入検討 |
| 中 | 維持 | - |
| 低 | 見直し | - |

#### 7.1.6 出力JSON構造

最終的に生成される`md_plan_final.json`は以下のキーで構成される。

| キー | 内容 |
|---|---|
| meta | メタデータ（分析日、対象期間等） |
| executive_summary | エグゼクティブサマリー |
| affinity | 相性ショップランキング |
| category_kaimawari | カテゴリ別買い回り傾向 |
| category_price_matrix | カテゴリ×価格帯マトリクス |
| internal_performance | 自社カテゴリ別パフォーマンス |
| md_proposals | 具体的なMD提案 |

#### 7.1.7 設定値

**価格帯定義 (PRICE_BANDS)**: 6段階（〜1,999円、2,000〜3,999円、4,000〜5,999円、6,000〜7,999円、8,000〜9,999円、10,000円〜）

**自社ブランド除外リスト (OWN_BRANDS)**: MONO-MART、EMMA CLOTHESなど22ブランドを含むリスト

#### 7.1.8 注意事項

- 各データソースのファイルパスは実行前にスクリプト内の定数を手動更新する必要がある
- 買い回りデータExcelはファイルサイズが大きいため、1シートずつ処理する設計となっている
- 注文データCSVには列のずれ（ヘッダー28列に対しデータ29列）が存在する可能性があり、スクリプト内で正しい列マッピングを定義して対応している
- 注文データに含まれないブランド（例: ADRER）は内部環境分析がスキップされ、外部環境分析の結果のみから「新規参入」の提案が生成される

## 10. データ定義スキル

### 10.1 zozo-order-data — 全ショップ ZOZO 注文生データ参照定義

#### 10.1.1 目的・用途

株式会社MONO-MARTが運営する全7ショップのZOZOにおける注文生データ（2025年4月〜2026年4月）へのアクセス方法とデータ構造を定義するスキルである。他の分析スキル（zozo-md-plan、zozo-same-item-cross-buy等）が注文データを参照する際の共通定義として機能する。

**トリガー例**: 併売分析、顧客分析、売上分析、リピート分析など、注文レベルの生データが必要なあらゆる分析タスク

#### 10.1.2 データ構造

全ショップ共通で28列のCSV形式である。主要な列は以下の通り。

| 列名 | 内容 |
|---|---|
| ショップ名 | 注文が発生したショップ |
| 親カテゴリ | 商品の親カテゴリ |
| 子カテゴリ | 商品の子カテゴリ |
| ブランド品番 | 品番 |
| 販売価格 | 注文時の販売価格 |
| キャンセル | キャンセルフラグ |
| モール | 販売チャネル |

#### 10.1.3 データ格納場所

各ショップの注文データはGoogle Drive上の個別フォルダに格納されている。MONO-MARTのように月次または半月でファイルが分割されている場合もあれば、他のショップのように全期間一括のファイルの場合もある。ダウンロードには`rclone`コマンドを使用する。

#### 10.1.4 注意事項

- データ量が非常に大きいため、分析に必要なショップ・期間・列のみを限定して読み込むことを推奨
- キャンセルされた注文が含まれている可能性があるため、「キャンセル」列でのフィルタリングが必要
- MONO-MARTについては品番別・月別の集計済みサマリーCSVが別フォルダに存在するため、それで十分な場合はそちらを優先使用

## 13. 共通技術仕様

### 13.1 データエンコーディング

| エンコーディング | 対象ファイル |
|---|---|

| UTF-8 | 注文生データCSV |

### 13.2 Google Workspace連携

| 環境 | 方式 | 備考 |
|---|---|---|
| サンドボックス（開発） | gws CLI + rclone | ローカルファイルアクセス可能 |
| デプロイ（本番） | サービスアカウント + Google API | gws CLI使用不可、環境変数GOOGLE_SERVICE_ACCOUNT_KEYが必要 |

### 13.3 商品画像URL生成ルール

ZOZOの商品画像URLは以下のパターンで生成される。

```
https://o.imgz.jp/{商品コード末尾3桁}/{商品コード}/{商品コード}b_{カラーコード}_d.jpg
```

例: 商品コード `72406170` でカラーコード `032` の場合

### 13.4 CDNデータ管理

分析結果のJSONファイルはS3 CDNに配置され、フロントエンドから直接読み込まれる。データ更新時は以下の手順を踏む。
